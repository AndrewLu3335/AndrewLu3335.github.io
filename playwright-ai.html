Evidence before inference

The analyzer receives sanitized test errors, page state, navigation, failed requests, HTTP errors, redacted console evidence, and selected screenshots. Its output is explicitly treated as a probable diagnosis that an engineer must verify.

Test codeLocator, assertion, synchronization, or implementation issue.
ApplicationProduct UI, backend, or runtime behavior appears defective.
Test dataAccount state, fixture, or required data is invalid or unavailable.
EnvironmentBrowser, dependency, configuration, or infrastructure failure.
NetworkTransport evidence directly blocked the tested behavior.
Flaky behaviorIntermittent behavior requires reproduction and trend evidence.

06

Implementation Decisions

The framework keeps intelligence useful and bounded.

  1. Separate verdict from diagnosisDeterministic assertions remain the only pass/fail authority.
  2. Collect evidence only when neededFailure-specific capture reduces noise in successful runs.
  3. Sanitize before analysisURLs and common credential-like console values are removed or redacted.
  4. Require causal network evidenceAn element-not-found error alone cannot become a network classification.
